# Turn Ambition Into Architecture.

ChronoForge is a premium full-stack execution-intelligence platform built to help people **plan serious goals, simulate deadline pressure, detect burnout risk, track real progress, compare execution paths, and export structured timeline reports**.

It is not a to-do list.
It is not a habit tracker.
It is not a simple productivity dashboard.

ChronoForge is a system for answering one difficult question:

> **Can this ambition realistically survive the time, effort, capacity, and pressure required to complete it?**

Built independently over **680 hours** across **12 weeks / 3 months**, ChronoForge represents a complete product journey: concept, engine design, interface architecture, authentication, database persistence, dashboard intelligence, testing, CI, export workflows, and polished product presentation.

> Built independently while still in high school with a focus on systems thinking, execution design, product depth, and meaningful planning intelligence.

---

# 🌍 The Idea

Most people do not fail because they lack goals.

They fail because they cannot see the structure behind their goals.

They rarely know:

* how much effort the goal truly requires,
* whether their deadline is realistic,
* whether their weekly capacity is enough,
* how much recovery buffer they have,
* whether their plan is quietly drifting,
* whether the goal is achievable but unhealthy,
* which project deserves attention first.

## ChronoForge solves this.

It transforms ambition into a system of:

### Forecasting • Diagnosis • Execution • Comparison

# 🛤️ 12-Week / 680-Hour Build Journey

ChronoForge was not built as a quick weekend project.

It evolved through a complete 12-week product-building journey.

## Week 1 — Product Question

The project began with one core question:

**Why do people plan goals without understanding whether the timeline can survive execution?**

This week focused on defining the problem:

* vague goals collapse under pressure,
* deadlines are often chosen emotionally,
* effort is underestimated,
* recovery is ignored,
* burnout is invisible until it arrives.

The first version of ChronoForge was born from this idea.

# 🚀 Getting Started

## Clone Repository

```bash
git clone https://github.com/sumitdhara609/chronoforge.git
cd chronoforge
```

## Install Dependencies

```bash
npm install
```

## Environment Setup

Copy the example environment file:

```bash
cp .env.example .env.local
```

On Windows PowerShell:

```powershell
Copy-Item .env.example .env.local
```

Add your Supabase values:

```env
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
```

Never commit `.env.local`.

## Run Development Server

```bash
npm run dev
```

Open:

```txt
http://localhost:3000
```

# 🗄️ Supabase Setup

ChronoForge uses a `timelines` table connected to authenticated users.

Core fields:

```txt
id
user_id
goal_title
total_estimated_hours
available_hours_per_week
days_until_deadline
created_at
updated_at
last_exported_at
execution_status
progress_percentage
last_progress_updated_at
```

The table should use Row Level Security so users can only access their own timelines.

# 🔐 Security Notes

* `.env.local` is ignored.
* `.env.example` is safe to commit.
* Service-role keys must never be exposed through `NEXT_PUBLIC_*`.
* Timeline rows belong to authenticated users.
* Row Level Security protects private timeline data.
* The app uses Supabase authentication for private workspace access.
